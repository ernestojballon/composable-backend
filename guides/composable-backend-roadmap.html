Composable Backend — Improvement Roadmap

Filtered for serverless compatibility. Ordered by effort → impact ratio.

# Improvement Effort Impact What it is Why it matters
1 tsx watch dev mode Trivial High DX An npm script ("dev": "tsx watch src/index.ts") that auto-restarts the app on every file save, no build step needed. Cuts the feedback loop from ~5 seconds to instant. Every modern framework ships this out of the box. Without it, the framework feels dated on first impression.
2 Typed events (typed-emitter) Low High Safety A TypeScript-only wrapper around the existing EventEmitter that enforces type checking on event names and payloads at compile time. Zero runtime cost. Catches typos like "v1.lead.scroe" at compile time. Prevents an entire class of silent event-routing bugs that are impossible to debug at runtime.
3 OpenTelemetry integration Medium High Ops A @composable-backend/otel package that bridges the existing tracer (exec_time, round_trip, trace IDs) to the OpenTelemetry standard. The framework already collects performance metrics but they're stuck in console logs. OTel sends them to Jaeger, Datadog, or Grafana. The difference between “we think scoring is slow” and a flame graph showing exactly which task is the bottleneck. OTel SDK works natively in Lambda.
4 Auto-generate OpenAPI Medium High API DX A build-time tool that reads rest.yaml (URLs, methods, auth, timeout) and composable inputSchema/outputSchema (Zod) to produce an OpenAPI 3.1 spec. Frontend teams currently read YAML config files or ask the backend developer. Auto-generated /docs gives an interactive API explorer, client SDK generation, and contract testing — all from config that already exists. Pure build-time, no runtime dependency.
5 create-composable-app CLI Medium High Adoption A command-line tool (npx create-composable-app my-project) that scaffolds a working project with config files, a sample composable function, and a flow — ready to run. Today, getting started means cloning the example project and deleting what you don't need. A scaffolding CLI is the first thing new users interact with. Every successful framework (Next.js, Fastify, Hono, NestJS) has one. Without it, adoption depends on people being patient enough to read docs first.
6 Test harness package Medium High DX A @composable-backend/testing package with utilities like TestPostOffice (captures sent events), TestPlatform (in-memory registration), and expectEvent() assertions. The framework has excellent testability by design, but developers wire up Platform, AppConfig, and PostOffice manually in every test. The harness reduces 15-line setup to 2 lines. Also enables event spying (“was this event sent with these headers?”) which is currently impossible without reading logs.
7 Structured JSON logging Low Medium Ops Ensure the existing log.format: json compact mode outputs machine-parseable JSON with consistent fields (timestamp, level, service, trace_id). CloudWatch, Datadog Logs, and ELK auto-parse structured JSON. The framework already has a compact format, but log analytics tools need consistent field names to build dashboards and alerts. A small refinement that makes the framework production-observable on day one.
8 Transactional Outbox pattern Medium High Reliability When a composable must update state AND publish an event (e.g., save to DB then publish to Kafka), do both in a single DB transaction. Write the event to an “outbox” table; a relay process publishes it later. The #1 silent failure mode in event-driven systems: a crash between DB write and event publish loses data permanently. Works perfectly in serverless — the outbox is just a DB table, and the relay can be a scheduled Lambda. Could be outbox: true in kafka-adapter.yaml.
9 BFF pattern guidance Low Medium Arch Document how to use composable-backend as a Backend-for-Frontend layer — a dedicated backend per client type (web, mobile, partner API) that aggregates internal services. The framework already does BFF perfectly: per-client REST endpoints, flow aggregation, per-client auth. Just needs docs showing the pattern. Each BFF deploys as its own Lambda. No new code — pure documentation win.
